# 中1テストキット 認証・購入フロー設計

作成日: 2026-05-14

## 目的

この設計の目的は、購入者が迷わず学習を始められることと、運営側が購入フローを確実に検証できることを両立することです。

今回の問題は、購入前ログインを必須にしたことで、メール送信サービスの制限がそのまま購入フロー確認の停止要因になったことです。今後は「商品購入」と「ログインメール到達」を分けて扱います。

## 結論

このアプリの認証設計は、次の形にします。

1. 新規購入はログイン不要にする
2. Stripe Checkout でメールアドレスと決済を受ける
3. 決済完了後にユーザーを作成または更新する
4. 購入済みユーザーへログインリンクを送る
5. 既存ユーザーの再ログインはメールマジックリンクで行う
6. 検証用にはメールに依存しない安全なテストログイン手段を別に持つ

## 商品に合う理由

中1テストキットは、980円買い切り（期限なし）で使う定期テスト対策アプリです。ユーザーは長期運用の会員管理よりも、「買ってすぐ始める」ことを期待します。

そのため、通常のパスワード登録や複雑なアカウント作成は重すぎます。メールアドレスを本人確認と購入紐付けの軸にするのが最も軽い設計です。

ただし、購入前にメールログインを必須にすると、今回のようにメール送信が詰まった時点で購入検証まで止まります。したがって購入はログイン不要、購入後の利用開始にメール認証を使う設計が適しています。

## 本番フロー

### 新規購入

1. ユーザーが `/buy` を開く
2. メールアドレスを入力して「購入する」を押す
3. サーバーが Stripe Checkout Session を作成する
4. Stripe Checkout でカード決済を行う
5. Stripe Webhook `checkout.session.completed` を受ける
6. `customer_details.email` または Checkout 作成時のメールでユーザーを upsert する
7. `purchased_at` を設定する（買い切り永久。`expires_at` は使用しない）
8. ログインリンクをメール送信する
9. 成功画面で「購入完了」と「ログインメールを送信済み」を表示する

重要: 決済完了とメール送信は別処理として扱います。メール送信に失敗しても購入状態は失わせません。

### 既存ユーザーのログイン

1. ユーザーが `/login` にメールアドレスを入力する
2. サーバーがマジックリンクを送る
3. リンクのJWTを検証する
4. セッションCookieを発行する
5. 購入済みなら `/home`、未購入なら `/buy` へ送る

### 購入済みユーザーの再購入

購入済みユーザーが `/buy` を開いた場合、追加購入ボタンは出さず、ホームへ戻す導線を表示します。買い切りモデルのため、再購入は想定しません。

## 検証フロー

購入フロー確認は、メール送信に依存しない経路も用意します。

### Stripe購入テスト

1. 未購入メールで `/buy` を開く
2. Stripe test card で決済する
3. Webhook で `purchased_at` が入ることを確認する
4. `/checkout/success` が購入完了として表示されることを確認する
5. メール到達とは別に、DB上の購入状態を確認する

### テストログイン

本番公開ユーザーには見えない検証用ログインを用意します。

条件:

- `ENABLE_TEST_LOGIN=true` の環境だけで有効
- `TEST_LOGIN_SECRET` が一致した場合だけ使える
- 対象メールは `@takaapps.com` または `TEST_LOGIN_ALLOWED_EMAILS` に明示したメールだけに制限する
- 操作ログを残す（`test login issued`）
- 本番リリース前チェックで無効化を確認する

この経路は、Resendやメール到達性に依存せず、購入済み・未購入・期限切れの状態確認に使います。

API:

`POST /api/auth/test-login`

Body:

```json
{
  "email": "tester@example.com",
  "secret": "TEST_LOGIN_SECRET"
}
```

## メール送信設計

本番メールは Resend を使います。ただし、次の条件を満たすまで外部ユーザー向けには使いません。

- `takaapps.com` を Resend でドメイン認証済み
- `RESEND_FROM_EMAIL` は `noreply@takaapps.com` など認証済みドメインのアドレス
- `onboarding@resend.dev` は本番・購入検証に使わない
- API key は本番送信用として権限を確認済み
- 任意のGmail宛てにログインメールが届くことを確認済み

送信失敗時は、ユーザーには「メール送信に失敗しました」と表示し、運営側ログには Resend のエラー内容を残します。

## データ設計

現在の `users` は最低限の購入状態を持っています。

- `email`
- `purchased_at`
- `textbook_publisher`
- `test_date`

`expires_at` 列はレガシーで残っていますが、買い切り永久モデルへ移行したため新規書込・参照ともに行いません。

追加するべき購入監査データ:

- `stripe_checkout_session_id`
- `stripe_payment_intent_id`
- `amount_total`
- `currency`
- `purchase_email`
- `webhook_received_at`
- `email_send_status`

Stripe Webhook は同じイベントが複数回来る可能性があるため、`stripe_checkout_session_id` で冪等に処理します。

## 画面設計

### `/buy`

未ログインでも表示します。

表示内容:

- 商品名
- 価格
- 買い切り（期限なし）
- 対応範囲
- メールアドレス入力
- Stripe決済ボタン
- 利用規約、プライバシーポリシー、特商法表記

ログイン済みで購入済みの場合は「このアカウントは購入済みです」を表示します。

### `/checkout/success`

ログイン前でも表示します。

表示内容:

- 購入完了
- ログインメール送信状態
- メール再送ボタン
- メールが届かない場合の問い合わせ導線

この画面で「購入は完了しているがメールが届かない」状態を明確に切り分けます。

### `/login`

既存ユーザー向けの再ログイン画面です。新規購入者に最初に見せる画面ではありません。

## 再発防止ルール

次のルールを開発時の判断基準にします。

1. 購入フロー確認を始める前に、決済・Webhook・メールの依存関係を分けて確認する
2. ユーザーに別メールアドレスで試してもらう前に、メール送信サービスがその宛先へ送れる状態か確認する
3. 購入前ログインを必須にしない
4. 本番相当の検証には、未購入ユーザーを作れる手段を必ず用意する
5. 外部サービスのサンドボックス制限を本番仕様と混同しない
6. 「決済成功」と「ログインメール到達」を同じ成功条件にしない
7. 検証用ログインは本番公開前に無効化確認する

## 実装順序

1. `/buy` を未ログインでも使える購入入口に変更する
2. Checkout Session 作成APIを未ログイン対応にする
3. Stripe Webhook でメールアドレスからユーザーを upsert する
4. `/checkout/success` を追加する
5. 購入後ログインメールの再送APIを追加する
6. Resendの本番ドメイン認証を完了する
7. 検証用ログインを環境変数で gated にして追加する
8. 購入フローE2EチェックリストをREADMEまたはdocsに追加する

## 完了条件

この設計の完了条件は次の通りです。

1. 未ログインの新規ユーザーが購入できる
2. Stripe Webhook だけで購入状態が反映される
3. ログインメールが届けば、そのまま学習開始できる
4. メールが届かなくても、購入状態はDBで確認できる
5. テストログインで購入済み・未購入・期限切れを確認できる
6. 任意のGmail宛てに本番ログインメールを送れる
7. `t.nakaima@gmail.com` が購入済みでも、別の未購入ユーザーで購入フロー確認できる
